一、 名字的重大改变

Vue3 把这两个钩子改名了：

  • ❌ beforeDestroy —> ✅ beforeUnmount （卸载前）
  • ❌ destroyed —> ✅ unmounted （卸载后）

二、 Vue3 组合式 API（setup）中的生命周期

其他钩子全部加上了 on 前缀

在 setup 中使用生命周期，必须先从 Vue 中引进来，然后像调用普通函数一样使用它们：### Vue 生命周期钩子对比图 (Vue2 -> Vue3 setup)

阶段 传统写法 (Vue2 / 选项式) Vue3 组合式 (setup) 说明 / 常用场景
挂载前 beforeMount onBeforeMount 渲染 DOM 之前调用（此时还拿不到页面上的真实标签）
挂载后 mounted onMounted 🔥 最常用：DOM渲染完了，可以用来操作 Echarts 图表、获取真实 DOM 的高度等
更新前 beforeUpdate onBeforeUpdate 响应式数据更新，DOM 重新渲染前调用
更新后 updated onUpdated 响应式数据更新，DOM 重新渲染完成之后调用
卸载前 beforeUnmount (原 beforeDestroy) onBeforeUnmount 组件从页面上卸载/销毁前调用
卸载后 unmounted (原 destroyed) onUnmounted 🔥 最常用：AOP思想体现，用于资源清理，如清除定时器、断开 WebSocket 连接等

💡 补充提示：
Vue2 中的 beforeCreate 和 created 在 Vue3 的组合式 API 中被完全移除了。
因为 setup() 函数本身就是在这个阶段执行的，所以以前写在 created 里发 AJAX 请求的代码，现在直接写在 setup() 函数体内即可。

定时器：如果不清理掉会一直执行

定时器

定时器（setInterval 或 setTimeout）是挂载在浏览器的全局对象（window）上的。
如果你在一个组件里开启了定时器，当这个组件被销毁（比如用户跳到了另一个页面）时，定时器并不会自动停止。它会一直在后台默默运行，尝试去修改已经不存在的变量，最终导致页面变卡甚至程序崩溃。

  • 如果你用的是 setInterval（循环执行），对应的清除函数是 clearInterval(timer)。
  • 如果你用的是 setTimeout（延迟一次执行），对应的清除函数是 clearTimeout(timer)。

5. 计算属性(computed)

函数和计算属性的最大区别：计算属性有“缓存机制”，而函数没有！

  • 数据 age 变了，页面需要重新渲染更新。只要页面一刷新，模板里的 {{ getUpper() }} 就会被重新调用一遍。这很合理。

  • 但是没有修改 age，为什么 getUpper() 函数也执行了？因为你修改 name 导致了整个页面的重新渲染（DOM 更新）。普通的函数非常“傻”，只要页面有任何风吹草动引发重新渲染，它就会无脑地重新执行一遍

  • 而计算属性，有缓存，数据没有变化，页面再更新，它也不变化

6. 监听属性（watch）

监听与计算的不不同

1.核心定位与返回值不同

  • computed（计算属性）：重点是计算。它必须有一个返回值，用于根据现有数据派生出一个新数据。你在模板里使用它，就像使用普通数据一样。
  • watch（监听属性）：重点是动作。它不需要返回值，当监听的数据变化时，它执行的是一段业务逻辑或副作用（如发送网络请求、操作DOM、修改其他状态等）。
  1. 是否支持缓存（性能差异）
  • computed 支持缓存：只要它依赖的数据没有发生变化，多次访问计算属性都会直接从缓存中取值，不会重新执行计算逻辑。这对性能优化非常关键。
  • watch 不支持缓存：只要监听的数据发生了变化，回调函数就会无条件重新执行。
  1. 是否支持异步操作
  • computed 不支持异步：计算属性中如果有异步操作（如发送Ajax请求或setTimeout），它是无法监听到异步结果变化的，也无效。
  • watch 支持异步：这是watch最大的优势。当数据变化后，需要在异步操作完成后再执行某些逻辑，必须使用watch。
  1. 数据流向与适用场景（多对一 vs 一对多）
  • computed 是多对一,适用于一个属性受多个数据影响的场景。
    • 举例：购物车总价（受商品单价、数量、折扣等多个数据影响计算得出）。
  • watch 是一对多：适用于一个数据变化影响多个操作或其他数据的场景。
    • 举例：就像你刚才代码里的场景，分类名一变化，就要去向后端发请求获取新列表，这就是一个数据变化触发了一个复杂操作。

8. 组件通信

8.1 父子通信之父传子

通过自定义属性,绑定传信变量

8.2 父子通信之子传父

通过自定义事件

核心思路：

  1. 子组件：通过 emit(‘事件名’, 传过去的数据) 触发一个事件（打电话）。
  2. 父组件：在子组件的标签上，用 @事件名=”执行的函数” 来监听这个事件（接电话并处理）。

Vue3 规定 setup 函数接收两个参数，顺序是写死的：

  • 第一个参数：是父组件传过来的 props。
  • 第二个参数：是上下文对象 context（里面包含了 emit、slots、attrs 等）。

8.3 ref属性

**在 Vue3 中，**ref 获取 DOM 和组件对象的功能完全保留，但是用法和“规矩”发生了巨大的变化！

  1. 在 Vue3 的 setup 里，没有 this，所以彻底废弃了 this.$refs.xxx 的写法。
  1. 通过 ref 操作子组件

默认情况下，父组件拿到子组件对象后，里面是空的，什么都用不了！

Vue3 的新规矩：子组件必须“主动暴露（expose）”，父组件才能用！
