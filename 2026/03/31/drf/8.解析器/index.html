解析器原理

解析请求者发送过来的数据，如：json

为什么常用json

注意：关于QueryDict

对于原生django的疑惑

既然 POST 请求可以发送 JSON 数据，而 JSON 解析出来明明是普通的字典（dict），那你怎么能说 request.POST 返回的都是 QueryDict 呢？

核心在于：在原生 Django 中，如果前端发送的是 JSON 数据，这些数据根本就不会进入 request.POST！

彻底梳理一下这里面的机制：

  1. request.POST 只认“表单数据”

在 Django 的底层设计中，request.POST 被严格限定为只处理HTML 表单提交的数据。

  • 必须是 Content-Type: application/x-www-form-urlencoded
  • 或者是 Content-Type: multipart/form-data（上传文件）

因为 HTML 表单允许同名的多选框（比如勾选多个爱好 hobbies=reading&hobbies=sports），为了能装下同一个 Key 的多个 Value，Django 强制规定 request.POST 的类型必须是 QueryDict。

  1. 那 POST 过来的 JSON 数据去哪了？

如果前端通过 POST 请求发送了一段 JSON（Content-Type: application/json），Django 的反应是这样的：

  1. Django 发现这不是表单类型。
  2. Django 决定不管它，此时 request.POST 里面是空的，也就是一个空的 <QueryDict: {}>。
  3. 前端发来的真实 JSON 字符串，被原封不动地塞进了一个叫 request.body 的属性里（这是一个 bytes 字节流）。

如果你用的是原生 Django（没有用 DRF），你要拿 JSON 数据得这么写：

  1. DRF 的 request.data 是怎么回事？

因为每次都要去 request.body 里手动 json.loads 太反人类了，所以 Django REST Framework (DRF) 站出来解决这个问题。

DRF 对原生的 request 进行了升级，提供了一个全新的属性：request.data（注意不是 request.POST）。

当你在 DRF 中使用 request.data 时：

  • 如果前端发来的是表单，DRF 给你返回一个 QueryDict。
  • 如果前端发来的是 JSON，DRF 底层的 JSONParser 自动帮你解析，给你返回一个 普通的 Python dict！

总结:

  1. request.GET：永远是 QueryDict（因为 URL 参数和表单结构一样，可能有多值）。
  2. request.POST（原生）：永远是 QueryDict，但它里面只有表单数据，没有 JSON。
  3. JSON 数据：在原生 Django 里存在 request.body 中（解析后是普通 dict）；在 DRF 中存在 request.data 中（解析后也是普通 dict）。

所以，当你处理 JSON 数据时，接触到的确实是普通的 dict，而不是 QueryDict！

问题：

  1. 为什么用 json.load，不用 json.loads

在 Python 的 json 模块中，这两个函数有明确的分工：

  • json.loads(s) (load string)：
    • 它接收的是一个完整的字符串（或字节串）。
    • 这意味着，你必须先把所有数据从网络中读取出来，全部塞进内存，变成一个巨大的字符串，然后才能传给它。
    • 如果前端发来一个 50MB 的 JSON，你的服务器内存里瞬间就会多出一个 50MB 的字符串，极其消耗内存。
  • json.load(fp) (load file pointer)：
    • 它接收的是一个文件对象（或类似文件的流对象 stream），只要这个对象有 .read() 方法就行。
    • 它是流式读取的。它不会一次性把 50MB 全部读进内存，而是一边从流里往外抽数据，一边在内存里构建 Python 字典。
    • 这种方式极其节省内存，性能也更高。
  1. 实时流媒体翻译

Python 的 codecs 模块提供了一个极其强大的工具：流式解码器。它可以在“不破坏流式特性”的前提下，一边读取字节，一边把它翻译成字符串。
