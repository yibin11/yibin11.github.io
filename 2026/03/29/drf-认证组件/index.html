为了让你更透彻地理解这个过程，我们可以看一下 DRF 内部的“执行流程图”：

  1. DRF 的执行流水线

当一个请求到达 Django 视图时，DRF 的 APIView 会按顺序执行以下操作：

  1. dispatch(): 这是总入口。

  2. initial(): 初始化阶段，这里包含三大守门员：

    • 认证 (Perform Authentication)：确定你是谁（如果你在这里抛出异常，流程直接跳出）。
    • 权限 (Check Permissions)：确定你能不能看。
    • 限流 (Check Throttles)：确定你访问是不是太快了。

  3. 视图函数 (Handler)：如 get(), post() 等业务逻辑。

  4. 异常抛出后的“紧急跳出”机制

在 Python 中，raise 的作用是中断当前的执行路径。

当 authenticator.authenticate(self) 抛出 APIException 时：

  1. 代码运行到 raise 语句。
  2. 它不再继续向下执行 initial() 里的剩余代码（跳过权限检查和限流检查）。
  3. 它直接跳出 initial() 方法。
  4. 它跳出 dispatch() 方法。
  5. 最后，它被 DRF 的**全局异常处理器（Exception Handler）**捕获。
  6. 异常处理器把这个异常转换成一个 Response 对象（包含 401 或 403 状态码）。
  7. Django 把这个 Response 返回给客户端。

结论：视图函数中的业务逻辑代码根本没有机会被触碰。

  1. 特殊情况：为什么有时认证失败还能进视图？

这里有一个细节，可能会让你产生困惑，所以必须说明：

  • 情况 A：抛出异常（你刚才看到的源码）
    如果认证组件（如 Token 认证）发现你提供了 Token，但 Token 是无效的/过期的，它通常会 raise AuthenticationFailed。

    • 结果：中断执行，不进视图，直接返回 401。

  • 情况 B：没有提供凭证（返回 None）
    如果你根本没有在 Header 里传 Token，认证组件会执行 return None（而不是抛异常）。
    DRF 发现所有认证组件都返回了 None，它就会执行 _not_authenticated() 把你设为“匿名用户”。

    • 结果：认证通过（身份是匿名），流程继续。
    • 后续：紧接着的权限组件（Permission）会检查。如果你设置了 IsAuthenticated 权限，权限组件会抛出异常并拦截；如果你设置了 AllowAny，那么这种情况下会进入视图。

2.状态码

对于401与403的区别

特性 401 Unauthorized 403 Forbidden
核心含义 身份未验证 / 验证失败 身份已验证，但权限不够
针对对象 身份（Who are you?） 行为（What can you do?）
处理流程 发生在 Authentication 阶段 发生在 Permissions 阶段
客户端反应 引导用户去登录 / 刷新 Token 告诉用户“你没权访问此资源”
登录能解决吗？ 能（登录后就变成合法用户了） 不能（登录了也没权限，除非换个号）
